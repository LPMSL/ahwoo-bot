## 三、目前仍有缺口

### 3.1 尚未完成

1. **Google Calendar 名額查詢**
   - 顧客問日期時，仍無法直接回覆「有名額 / 已滿 / 需人工確認」

2. **圖片 / 附件處理**
   - 顧客傳設計圖、參考圖時，仍需人工判讀

3. **自取時間多輪確認**
   - 取貨時段與特定時間確認仍缺少狀態機流程

4. **長期跨對話記憶**
   - 目前只有 24 小時 session 記憶
   - 尚未建立 user_id 對應的長期訂購歷史 / CRM 記憶

5. **候補名單與取貨後回訪**
   - 尚未建立 waiting list
   - 尚未自動發送取貨後感謝 / 評價邀請

### 3.2 已有初版，但需補強

1. **未回覆警報**
   - 已能每小時掃描並推播
   - 但仍需補「去重機制」與更精準的營運規則

2. **跨對話記憶**
   - 已有 Redis session 記憶
   - 但仍不等於完整長期記憶

3. **FAQ 涵蓋率**
   - 已補強常見 FAQ 與致謝訊息
   - 但仍需持續用真實對話資料擴充關鍵字與測試案例

## 五、優先順序建議

### Phase 1（近期，1-2 週）
目標：先把「目前已經有的 Bot」變成穩定可營運版本

1. **Google Calendar 名額查詢**
   - 建立 `calendar_handler.py`
   - 讓日期詢問從「一律填表等待」進化成「先自動查詢」

2. **未回覆警報補強**
   - 避免重複提醒同一批對話
   - 補上「已提醒」或「最後提醒時間」欄位 / 狀態

3. **緩衝回覆優化**
   - 當 `needs_human=true` 時，統一回覆更自然的等待訊息
   - 降低顧客感受上的等待焦慮

4. **建立意圖驗證集**
   - 從真實對話抽樣 100-200 筆
   - 人工標註正確意圖、是否該轉人工、理想回覆方向
   - 作為之後改 prompt / keyword / FAQ 的回歸測試基準

### Phase 2（1 個月）
目標：提高自動化涵蓋率

1. **長期記憶**
   - 在 Google Sheets 或獨立資料表建立 user_id → 最近訂購資訊 mapping

2. **候補名單**
   - 日期額滿時，讓 Bot 可記錄候補意願

3. **外燴流程深化**
   - 讓外燴詢問不只停在 FAQ
   - 補強外燴資訊收集欄位與通知格式

4. **取貨後自動感謝**
   - 建立取貨完成後的追蹤訊息

### Phase 3（2-3 個月）
目標：由客服效率提升，延伸到轉換率與回購經營

1. **數據 dashboard**
   - 每週追蹤對話量、自動回覆率、人工轉接率、表單完成率

2. **沉睡追蹤者再喚醒**
   - 設計新品 / 活動推播流程

3. **評價 / 照片蒐集**
   - 建立取貨後邀請分享與 UGC 蒐集流程

## 八、Codex 建議第一個技術任務

```text
整合 Google Calendar API

輸入：
- 顧客想預約的日期（從對話中擷取）

邏輯：
- 查詢該日 14:00-20:00 的訂單事件數量
- 根據上限 N 回傳：
  - True：可接單
  - False：已額滿
  - None：日期無效或需人工確認

輸出用途：
- 提供給 claude_handler.py / 主流程判斷日期詢問的回覆策略

檔案位置：
- Ahwoo_chatbot/handlers/calendar_handler.py（新建）
```
